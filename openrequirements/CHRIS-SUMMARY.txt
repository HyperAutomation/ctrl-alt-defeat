## TOP 5 WORKSHOPS RECOMMENDED
================================================================================

Priority: CRITICAL (all five require immediate scheduling)

WS-001: Swarm-Aware SORA Assessment Framework
  Feature: REQ002 (SORA Assessment for Swarm)
  Participants: CAA SORA assessor, applicant systems engineer, autonomy engineer, test engineer
  Duration: 3 hours
  Goal: Define "swarm-aware" operationally and map swarm-specific evidence requirements
  Key questions resolved:
    - What distinguishes swarm SORA from single-aircraft SORA?
    - What swarm failure modes must SORA address?
    - How do interdependencies affect SAIL levels?
  Outputs: Swarm SORA methodology, evidence checklist, SAIL assignment criteria

WS-002: Automatic Containment Design Specification
  Feature: REQ005 (Geofence/Containment Enforcement)
  Participants: CAA airspace regulator, applicant safety engineer, autonomy engineer, avionics engineer, test engineer
  Duration: 4 hours
  Goal: Define geofence tolerance, fail-safe mechanism, design standard, environmental factors, verification approach
  Key questions resolved:
    - What is physical geofence boundary tolerance (+/- X meters)?
    - What is fail-safe if onboard logic fails?
    - What design standard applies (DO-254 DAL B, DEFSTAN)?
    - What environmental factors affect containment (GPS drift, wind)?
    - How is containment verified (Monte Carlo, flight test)?
  Outputs: Containment design spec, geofence tolerance design, verification plan

WS-003: Meaningful Human Control & Crew Workload
  Feature: REQ010 (Human Oversight and Crew Fitness)
  Participants: CAA human factors inspector, applicant crew lead, test pilot, autonomy engineer, human factors specialist
  Duration: 3.5 hours
  Goal: Define meaningful human control with measurable criteria
  Key questions resolved:
    - What situational awareness frequency and display comprehensiveness required?
    - Which decisions require explicit crew authorization?
    - What is acceptable takeover time (<=X seconds)?
    - What workload limits ensure crew can maintain control?
  Outputs: Meaningful human control definition, situational awareness spec, workload assessment criteria

WS-004: Link Loss Survivability & C2 Assurance
  Feature: REQ006 (C2 Link and Spectrum Assurance)
  Participants: CAA spectrum specialist, applicant communications engineer, test engineer
  Duration: 2.5 hours
  Goal: Define time-to-loss threshold, survivable action, crew role
  Key questions resolved:
    - How long can link be lost before action triggered?
    - What is onboard action (Return-to-Launch, hold, safe descent)?
    - Are C2 loss and inter-vehicle loss independently survivable?
    - Can crew intervene to override default action?
  Outputs: Time-to-loss and action spec, link survivability verification plan

WS-005: Detect-and-Avoid Predictable Resolution
  Feature: REQ007 (Detect-and-Avoid Performance)
  Participants: CAA DAA assessor, applicant autonomy engineer, test engineer
  Duration: 3.5 hours
  Goal: Define predictable conflict resolution mechanism for cooperative and non-cooperative encounters
  Key questions resolved:
    - What avoidance maneuver is preferred (climb, turn, descent)? Why?
    - How does swarm coordinate during DAA (formation or independent)?
    - Can crew override DAA decisions?
    - How is DAA compliance verified (DO-365/F3442)?
  Outputs: DAA maneuver selection logic, swarm coordination rules, verification plan

## TOP 3 COLLABORATION GAPS
================================================================================

Severity: CRITICAL (all three directly block trial approval)

GAP #1: Swarm-Aware SORA Evidence Undefined (REQ002)
  Root cause: SORA assessor + applicant + autonomy engineer never agreed on what "swarm-aware" means
  Impact: Two assessors can defensibly reach different SAIL levels for identical systems -> inconsistent authorizations
  Dorothy-1 finding (confidence 10/10): Term used but never defined
  Solution: WS-001 (all-team workshop) to operationalize swarm-aware evidence requirements
  Ubiquitous language entry: "swarm-aware" (proposed status, requires workshop agreement)

GAP #2: Meaningful Human Control Lacks Measurable Criteria (REQ010)
  Root cause: Human factors inspector + applicant crew + autonomy engineer never agreed on what "meaningful" means
  Impact: Assessors cannot evaluate whether crew evidence is sufficient; different standards across trials
  Dorothy-2 finding (confidence 10/10): Term is subjective and borrowed from LAWS debate; no operational thresholds
  Solution: WS-003 (all-team workshop) to define situational awareness, authority, takeover time, workload limits
  Ubiquitous language entry: "meaningful human control" (proposed status, requires measurable criteria)

GAP #3: Containment Design Criteria Absent (REQ005)
  Root cause: Safety-critical mechanism specified as outcome need ("automatic containment") without design criteria
  Impact: Developer cannot implement; different implementations will have inconsistent safety levels
  Dorothy-4 finding (confidence 10/10): Five critical subcriteria undefined:
    1. Geofence boundary tolerance (no tolerance specified)
    2. Fail-safe mechanism (if onboard logic fails, what happens?)
    3. Design standard (DO-254? DEFSTAN? Custom?)
    4. Environmental factors (GPS drift, wind, multipath not addressed)
    5. Verification approach (Monte Carlo? Flight test? Simulation?)
  Solution: WS-002 (all-team workshop) to specify design and verification criteria
  Ubiquitous language entry: "containment" (proposed status, requires design specification)

## KEY INSIGHTS
================================================================================

1. ISOLATION SIGNALS (12 findings):
   Specifications were written without collaborative input from all stakeholder perspectives.
   - Technical requirements specify outcome need but not mechanism
   - Safety-critical terms (defensibly, meaningful, containment) defined without regulator/applicant agreement
   - Missing perspectives: autonomy engineers (for swarm-aware, emergent behavior), human factors specialists (for workload, take-over), test engineers (for verification approaches)

2. TERMINOLOGY CONFLICTS (3 findings):
   Same term used with different meanings across requirements:
   - "complete" means different things in REQ001 (application intake) vs REQ012 (incident investigation)
   - "defensibly" assessing SORA (REQ002) with no criteria
   - "worst-credible" behavior boundary (REQ003) undefined

3. MISSING PERSPECTIVES:
   - Autonomy engineers: Not consulted on swarm-aware SORA, emergent behavior, decision-making architecture
   - Test engineers: Not consulted on verification approaches for containment, link loss, contingency
   - Investigators: Not consulted on evidence completeness, MOR triggers, data preservation
   - Human factors specialists: Not consulted on meaningful control, workload, take-over
   - Environmental experts: Not consulted on environmental impact (REQ019)
   - Security officers: Not consulted on classified evidence handling (REQ018)

4. REGULATORY GAPS:
   Four critical features (REQ017-REQ020) are regulatory obligations but lack user stories:
   - CAP 1616 (public engagement) - implied but unspecified
   - CAP 722H / JSP 440 (classified evidence) - implied but unspecified
   - Environmental regulations - implied but unspecified
   - Institutional learning / audit trail - implied but unspecified

5. VIBE SCALES ALIGNMENT:
   Vibe requirements (Tom Gilb's Planguage) operationalize goals but surface gaps:
   - PackCompleteness scale (REQ001) operationalizes "complete" for application intake
   - Identifies same ambiguity Dorothy-8 found (same term, different meanings in REQ012)
   - Scales provide quantitative targets that user stories lack
